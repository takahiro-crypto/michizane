1. 背景と方法

  • 狙い：ジョブカン・AppSheet 等のSaaS業務を、連携ではなく「削減＋Claude Code内製」で Tatsuya に集約する。
  • データ：社員別 操作ログ分析レポート301件（Lanscope・2026-04）。各レポートの自動化提案の月次期待効果（「月 約N分短縮」）を機械抽出・合算（範囲は下限採用＝保守的）。
  • 換算前提：想定人件費¥3,000/h（間接費込み・調整可）、1人月=160h。
数値の性格：労務削減はモデル推計であり実測ストップウォッチではない。カテゴリ間で一部重複しうる（合計は上限的）。1か月サンプル。SaaSライセンス削減額は各社の契約席数×月額が必要で、本書では方式と参考値のみ示す。意思決定は桁感で行い、精緻化は実契約データで別途。

2. 主要な発見 — 「手動巡回」と「SaaSの重複」

最多の行動は、承認・ステータス確認のためだけに各サービスを開く手動巡回（全社 月 約43万件）。加えて、同種SaaSの重複契約の兆候（会計、課題管理など）が見える。前者は内製フロントで、後者は統合廃止で削れる。

3. SaaS仕分けフレーム（3分類）

KEEP 残す＋束ねる

専門性・法令・規模で内製が非現実的。Tatsuyaから連携で叩く。（例：Salesforce, 奉行, SmartHR/HRMOS, Zendesk, Splunk）

REPLACE 内製する

「フォーム＋承認＋ステータス」の汎用業務／API貧弱／少席で割高。Claude Codeで作り替えSaaS解約。（例：Jobcan WF承認, 受付, 社内独自）

CUT 廃止・統合

同種SaaSが重複／低利用。1本化・停止。（例：会計ツール重複, 課題管理二重）

5. Claude Code が「開発エンジン」（実証済み）

内製化の手段はまさに Claude Code。SaaSを買う代わりに、業務システムを数分〜数日で作る。本提案の裏付けとして、以下は既に Claude Code で構築・本番稼働している。

  • オートクリスタル自動生成コア：業務の説明文から入力画面を自動生成 → 従来SaaSへ転記＋AIready構造化（設計/運用を分離し、社員はロック済み画面を使うだけ）。
  • 承認フローMVP：承認待ちを能動表示 → AI要約つきカードで承認/差戻し → 自然言語で状況照会 → AIready蓄積。Jobcan WFのリプレイス試作。
  • アダプタ層：コネクタ差し替えで「残すSaaSは連携／内製は自前実行」を1つの仕組みで扱う。

＝「Claude Codeで業務システムを内製できる」証拠そのもの。内製の限界費用は従来の受託開発より桁で低い。

7. 安全に回す運用モデル ― 夜間SKYDIV × 朝ホーリー

内製リプレイス・道真の検知・設計書の自動生成を「いつ・どこで・誰の責任で」動かすかの運用設計。夜間は隔離VDIでAIが読み取り・下書き、朝は人が確認・判断。統制を効かせたまま24時間活用する。

🌙 夜間（社員不在）― SKYDIV（隔離VDI）

空きデスクトップでAIが自動実行
  • プロセスマイニング → 設計書のたたき台（SKYSEAログから as-is 業務を抽出）
  • 道真で突合・スコアリング → 不正検知の候補（持ち出し・退職者リスク）
  • 読み取り・下書きのみ。不可逆操作（送信・送金・同意）はしない

☀️ 朝（出社後）― ホーリー（タツヤ）

朝ダイジェストで結果を差し戻し、人が確認・判断
  • 設計書ドラフトをレビュー・確定（規程・承認ルートは人が補う）
  • 不正検知の候補を管理部門・監査が点検
  • 不可逆な対応（送信・処分・確定）は必ず人が判断
ガードレール：隔離（SKYDIV）／ allowlist ／ 反復回数の上限 ／ 全操作のログ ／ 不可逆操作は人 ／ 「AIはセンサー、判断は人」
AIガバナンス ― 道真でAI自身も監視（IRI）：夜間に自律実行するAIエージェントの操作ログも道真AI連携で突合・スコアリングし、allowlist逸脱・想定外操作・過剰な反復を検知。社員だけでなくAIの挙動も監視対象にする二重の統制（監視は IRI が担う）。
実装のポイント：GUI操作が要る所だけ computer use。ログ突合などは通常のバッチ処理で十分。夜間はVDIの空きシートを活用するのでコスト増を抑えられる。

この運用モデルにより、オートクリスタルの内製リプレイス／道真の検知／設計書の自動生成を、統制を効かせたまま「夜間に仕込み → 朝に人が確定」で回せる。今回の操作ログ分析（SaaS巡回の実測）と同じSKYSEA/操作ログを、夜間バッチの入力としてそのまま活用できる。

10. Phase 1 の実現性と留意

Jobcan WF APIは読み取り専用（GET /requests/ のみ。申請作成・承認・差戻しのAPIは無い）。→ だから「連携」ではなく「Claude Code内製リプレイス」が正解。読み取りAPIは能動通知・NL照会に活用し、承認操作は内製フローで持つ。
統制：誤承認は重大。承認権限・代理承認・職務分掌を正しく写像し、監査ログ・最終承認は人間。認証情報の安全な保管。
