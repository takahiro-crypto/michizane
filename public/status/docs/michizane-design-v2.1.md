## Opusレビュー・確定（v2.1）

**総評**：Fable 5 改訂版 v2.0 を道真の**正式設計として採用**。Opus起案の構造的な穴（①層順の逆立ち②判断時スコアが再現できず監査証跡にならない③「匿名化」は法的には仮名化④「本文不使用」と「粉飾=Slack削除」の矛盾）を的確に是正し、名寄せ設計・4軸スコアの式・閾値のアラート予算方式・誤検知運用ループまで実装粒度に到達している。**事実誤りは検出されず**、数値・係数はすべて「要確認/仮」で明示されており過剰な断定もない。

**Opusが追加する"要確認"（運用前に潰す）**：
1. **e-Realty版ホーリーの実体アクセス**：現在このPC/リポジトリにあるのは Eltes版 `eltes-holly`（タツヤ）。OBICを内包する **e-Realty版ホーリー（賃貸管理システム）のコード・データ・エクスポート経路は別物**で、前倒しの前提として実体と取得手段の確認が要る。
2. **Slackの利用・メタ取得可否**：粉飾の「削除メタ急増」は例示。e-Realty/Eltesで実際にSlackが使われ、削除・編集メタをログとして取得できるか要確認（無ければ他の削除/改ざんシグナルで代替）。
3. **person_key対応表の"監査部門別系統"**：組織・職務分掌の前提。現体制で「道真本体・AI推進室から復元不能」を実現できるか（属人化リスク#1と直結）を要確認。
4. **法人番号の付与率**：counterparty名寄せの生命線。SFA/会計での付与状況を Phase1 で**まず実測**し、循環取引Detectorの有効化ゲートに使う。
5. **数値は仮のまま運用に入らない**（Fableの明記を踏襲）。閾値・重み・保持期間・アラート予算・損失係数は監査/法務/情シス合意で確定してから稼働。

**確定**：本書（v2.0本文＋本注記）＝道真の正式設計 **v2.1**。以降の実装はこれを基準とする。

## 改訂サマリ（Fable 5）

Opus版 v1.0 からの主要変更点：

- **層構造を修正**：クロス検知が「蓄積層の上・検知層の中」で曖昧だったのを、**A供給 → B蓄積・正規化（名寄せ含む）→ C検知（Detector＋クロス）→ D提示・統制 → E価値可視化**に並べ直し、**層間I/F（データ契約・NormalizedEvent・Finding・Case）を明文化**。
- **自己矛盾を解消**：「チャット本文原則不使用」と「粉飾＝仕訳×Slack削除」の矛盾を、**Slackはメタデータ（削除イベント・頻度・時刻）のみ使用、本文は不使用**と規定して解決。
- **「scoreは保存せず再算定」を撤回・二重化**：監査証跡には**判断時点のスコアのスナップショット（エンジンver付き）を不変保存**し、現在ビューは再算定。監査可能性とエンジン進化を両立。
- **突合キー（名寄せ）を設計**：人＝person_key（HMAC仮名化＋監査部門限定の対応表）、取引先＝法人番号優先のcounterparty_key、期間＝会計カレンダー正規化。
- **Detector共通インターフェイスと4軸スコアの式・閾値決定手順（アラート予算方式＋シャドーモード）を実装粒度で定義**。
- **誤検知の運用ループを新章化**：トリアージ状態機械、FP理由コード、抑制ルール（期限付き）、ルール別precision proxyでの自動降格。
- **プライバシーを現実に合わせて強化**：ハッシュ化は「匿名化」でなく**仮名化**と正しく位置づけ、少人数部署の再識別リスク・保持期間・従業員周知（就業規則根拠）を明記。
- **LLM補助要約へのプロンプトインジェクション対策**を追加（証拠テキストはデータ扱い・要約なしでも原本閲覧可能・出力は「参考」ラベル）。
- **「防いだ損失額」の算定ルールを保守化**：実測と推計を分離し、係数を明示。攻め側の実測値と混ぜない。
- **ロードマップに各Phaseの完了基準（exit criteria）**を設定。クリティカルパスは検知ロジックでなく**データ供給**であると明示。
- **e-Realty（ホーリー）の賃貸管理システムには会計システムOBIC（奉行）のデータが既に入っている**事実を反映：e-Realty側の粉飾・循環取引は外部SaaS連携を待たず**ホーリーからのエクスポート先行で早期着手可能**とし、ロードマップを二系統（e-Realty前倒し／Eltes=タツヤ側は要連携）に分離。
- 鳥獣（熊）Detectorは道真コアの対象外とし、「プラガブルI/Fの実証事例」として位置づけを整理。
- **弱点・未解決論点を対処案つきで率直に列挙**（属人化、OSINTの名誉毀損リスク、夜間AI自身のインサイダー化 等）。

## 0. 一言サマリ

道真は **「決定論の検知エンジン ＋ プラガブルDetector ＋ クロス検知 ＋ 価値可視化」** の4点セット。データはホーリー（申請/承認/QA）・オートクリスタル（SaaSアダプタ）・SKYSEA/操作ログ・会計/SFA・IRI実績から供給し、夜間はSKYDIV（隔離VDI）で読み取り・下書きのみ自律実行、朝はホーリー経由で人が確定する。守った価値は「防いだ損失額」として保守的に算定し、経営に見える化する。

## 1. ビジョン・原則（検知の憲法）

- **目的**：内部不正・情報漏洩・取引リスク・AIの逸脱を早期検知し、被害を未然に圧縮する。
- **原則（全Detector・全フェーズで不変）**：
  1. 一次検知は**決定論ルールのみ**。LLM・確率モデルを検知経路に入れない。
  2. すべての候補に**根拠（evidence）**を添える。根拠のない加点は禁止。
  3. **不可逆操作（是正・処分・通報・取引停止）は人**。道真は候補提示まで。
  4. 出力は**「候補」であって判定ではない**。表示文言も「〜の可能性」「要確認」に統一（断定表現の禁止をUI規約に含める）。
  5. LLMは**検知済み事案の承認者向け補助要約のみ**（§6参照）。
- **ELTES本業との接続**：内部脅威（UEBA/IRI）＋ソーシャルリスク（OSINT）＋内製化（オートクリスタル）を「検知」の一本の軸で束ねる。道真はIRIの検知ノウハウの製品化母体であり、社内実証（ドッグフーディング）が営業資産になる。

## 2. 全体アーキテクチャ（5層・改訂）

```
[E] 価値可視化層   michizane：攻め(AX診断/SaaS削減実測) ＋ 守り(防いだ損失額)
      ▲ I/F: ValueMetric（Case確定結果の集計。個人特定情報を含まない）
[D] 提示・統制層   検知コンソール／Caseトリアージ／承認・是正はホーリーWF再利用／監査ログ
      ▲ I/F: Finding → Case（グルーピング済み候補＋根拠＋スコアスナップショット）
[C] 検知層         Detector群（決定論ルール）＋クロス検知（相関ルール/グラフ）＋4軸スコア
      ▲ I/F: NormalizedEvent（名寄せ済み正規化イベント）＋RefData（基準値・マスタ）
[B] 蓄積・正規化層  AIready基盤：取込→スキーマ検証→名寄せ(person/counterparty/period)→時系列蓄積
      ▲ I/F: DataContract（ソースごとにスキーマ・鮮度・粒度・PII区分を契約化）
[A] データ供給層    ホーリー(申請/承認/QA。**e-Realty版は賃貸管理システム内にOBIC(奉行)会計データを含む**)
                    ・オートクリスタル(SaaSアダプタ: 奉行/MF/SFA/Zendesk等——タツヤ(Eltes)側会計はこちら経由)
                    ・SKYSEA/Lanscope操作ログ・BigQuery/IRI実績・Slack(メタデータのみ)
```

**Opus版からの変更理由**：旧構成では「蓄積・照合層（C）」が検知エンジン（B）の上にあり、クロス検知の置き場所が曖昧だった。クロス検知は名寄せ済みの蓄積データを前提とするため、**蓄積を検知の下に**置き、単発Detectorとクロス検知を同じ検知層（C）に同居させる。これで「Detectorの出力（Finding）もまた検知層内でクロスの入力になる」再帰構造が素直に書ける。

### 2.1 層間インターフェイス定義

- **DataContract（A→B）**：ソースごとに①スキーマ（バージョン付き）②供給方式（手動エクスポート→read-only接続→API、の段階）③鮮度SLA（日次/週次）④PII区分（本文含む/メタのみ/仮名化済み）を文書化。スキーマ変更はcontract改訂として扱い、無告知変更で検知が黙って壊れる事故を防ぐ。
- **NormalizedEvent（B→C）**：共通イベントスキーマ（§3）。
- **Finding / Case（C→D）**：Detector出力（§4）。同一subject×近接期間のFindingはCaseに束ね、コンソールにはCase単位で出す（アラート洪水の防止）。
- **ValueMetric（D→E）**：確定Case（TP）のみから算定。個人を特定できる情報はE層に上げない。

## 3. 共通データモデルと名寄せ（クロス検知の土台）

クロス検知の成否は検知ロジックではなく**突合キーの品質**で決まる。ここを最初に固める。

### 3.1 NormalizedEvent スキーマ（v1）

```
NormalizedEvent {
  event_id        : UUID
  source          : "holly" | "autocrystal.buggyo" | "skysea" | "sfa" | "slack_meta" | ...
  source_ref      : 元レコードID（原本参照用。B層内のみ保持）
  event_type      : "expense.apply" | "expense.approve" | "journal.post" |
                    "file.export" | "login" | "message.delete" | ...
  occurred_at     : timestamptz（JST正規化。ソース側TZの扱いをcontractに明記）
  actor           : person_key（仮名化済み。§3.2）
  target_person   : person_key?（承認者・対象者など）
  counterparty    : counterparty_key?（§3.3）
  amount          : numeric?（JPY正規化。外貨はレート日付つき換算、要確認）
  period          : accounting_period?（§3.4）
  attrs           : jsonb（種別固有属性。スキーマはevent_typeごとに定義）
  ingested_at     : timestamptz
  schema_version  : int
}
```

- **本文系フィールドは持たない**。Slackは `message.delete` 等のメタイベントのみ。領収書画像等の原本はB層の隔離ストアに置き、イベントには参照のみ。

### 3.2 人の名寄せ：person_key

- `person_key = HMAC-SHA256(salt, 社員ID)`。**これは匿名化ではなく仮名化**（§10で扱う）。
- 社員ID⇔person_keyの**対応表とsaltは監査部門管理の別系統**に置き、道真本体・AI推進室からは引けない。復元は監査部門の承認操作（ホーリーWF）を経由し、操作自体を監査ログに残す。
- 複数ソースの社員識別子（ホーリーのuser_id、SKYSEAの端末ユーザー、会計の起票者コード、Slackのuser_id）は**人事マスタを正とするIDマッピングテーブル**で社員IDに解決してからHMAC。マッピング欠損は検知スキップではなく**「名寄せ不能」イベントとして品質メトリクスに計上**（欠損が多いソースはクロス検知に使えないため、供給改善の優先度根拠になる）。
- 退職者・異動は人事マスタの有効期間つきレコードで扱う（退職者Detectorの前提）。

### 3.3 取引先の名寄せ：counterparty_key

- 第一キー＝**法人番号**（国税庁公表。SFA・会計双方に付与するのが理想。未付与データが現実には多い、要確認）。
- 法人番号がない場合：正規化名称（株式会社等の除去・全半角/異体字正規化）＋住所での突合。**曖昧一致は自動確定せず「候補ペア」として人がマスタ登録**（名寄せ自体を誤ると循環取引検知が誤爆するため、ここは人の確定を挟む）。
- 個人事業主・海外取引先の扱いは未解決論点（§14）。
- **e-Realty固有**：ホーリー（賃貸管理システム）内のOBIC仕訳・取引先マスタ・物件を突合対象に含める。賃貸管理では取引の実体が物件に紐づくため、counterparty×periodに加えて**property_key（物件）を第3の突合軸**として持つ（例：同一物件をめぐる仕訳×申請×承認のクロス）。物件マスタはホーリー内で一元管理されており名寄せコストが低い——e-Realtyが粉飾・循環取引の先行実装地になる根拠のひとつ。

### 3.4 期間の正規化：accounting_period

- 会計カレンダー（期首・締め日・四半期）をRefDataとして一元管理し、`occurred_at`から`period`を導出。「期末集中」「締め直前駆け込み」系ルールはすべてこのカレンダー基準で書く（各Detectorが独自に日付計算をしない）。

## 4. 検知エンジン（コア）

### 4.1 Detector共通インターフェイス

```
interface Detector {
  id: string                 // "expense_fraud", "osint_counterparty", ...
  version: semver            // ルール変更ごとに上げる
  subscribes: EventType[]    // 必要イベント種別を宣言（供給側の依存が明示される）
  requires_ref: RefKey[]     // 基準値・マスタへの依存を宣言
  detect(events, ref) -> Finding[]   // 純関数。副作用なし・外部通信なし
}

Finding {
  finding_id, detector_id, detector_version, rule_id
  subject      : person_key | counterparty_key
  window       : [from, to]
  evidence[]   : { event_id[], description, value }   // 必ず元イベントに遡れる
  axes         : { deviation, impact, urgency, recurrence }  // 各0.0–1.0
  score_snapshot : { score, rank, engine_version, computed_at }  // 不変
  status       : triage状態（§8）
}
```

- `detect`を**純関数**に縛るのが要点：再実行で同じ入力から同じFindingが出る（テスト可能・監査で再現可能・SKYDIV夜間バッチと日中再計算が同一コード）。
- 新Detector追加＝このI/Fの実装＋DataContract確認＋シャドーモード投入（§8.3）。鳥獣（熊）Detectorはこのプラガビリティの実証事例であり、道真（内部脅威）コアのスコープ外として別カタログ管理。

### 4.2 4軸スコア：式と正規化

各ルールは発火時に4軸へ**根拠つき寄与**を出す。軸値は0–1に正規化：

- **逸脱度 deviation**：本人/同部署ピア集団の過去分布に対する分位点。例：申請額なら `dev = percentile_rank(amount, 同職位・過去12ヶ月分布)` を0–1化。分布が薄い（新入社員等）場合は全社分布にフォールバックし、その旨をevidenceに残す。
- **影響度 impact**：金額・件数・データ量を対数スケールで0–1化。`impact = min(1, log10(amount / floor) / log10(cap / floor))`。floor/capはDetectorごとにRefDataで設定（初期値は仮置きし要確認、§4.3の手順で調整）。
- **切迫性 urgency**：直近性＋不可逆性。例：退職予定日まで残りN日、期末締めまで残りN日を指数減衰で0–1化。
- **反復性 recurrence**：同種Findingの過去発生回数を飽和関数で0–1化（`rec = n / (n + k)`、k=3程度から開始）。

合成：

```
score = 100 × Σ w_i × axis_i    （Σw_i = 1、初期値 w = 0.35/0.30/0.20/0.15）
rank  = 高(score ≥ T_high) / 中(≥ T_mid) / 留意(それ未満)
```

重みと閾値T はDetectorごとのRefDataであり、コードに埋め込まない。

### 4.3 閾値の決め方：アラート予算方式

閾値を「正しさ」から先に決めようとすると決まらない。**運用者が捌ける量から逆算する**：

1. **アラート予算を先に決める**：監査部門が1日にトリアージできるCase数（例：高5件/中15件。実数は監査部門と合意、要確認）。
2. **コールドスタート**：過去データ（ホーリーエクスポート等）にDetectorを一括適用し、スコア分布を取得。予算内に収まる分位点をT_high/T_midの初期値にする。
3. **チューニング**：§8のフィードバック（TP/FP判定）が溜まったら、ルール別precisionを見て重み・閾値を月次で調整。調整はRefData変更＝バージョン管理し、変更前後の分布比較を記録する。

### 4.4 スコアの保存方針（Opus版から変更）

- Opus版「scoreは保存せずrecordを保存→参照時に再算定」は、**判断時点に人が見たスコアが後から再現できず監査証跡として不成立**。
- 改訂：**検知時点のscore_snapshot（エンジンver付き）をFindingに不変保存**。コンソールの現在ビューは最新エンジンで再算定した値を併記（「検知時: 78 / 現行エンジン: 71」）。人の判断記録は常にスナップショット側に紐づく。

## 5. クロス検知

単一ログで確証しない、が道真の核。実装は**相関ルール**と**グラフ検知**の2種。

### 5.1 相関ルール（Finding×Finding / Finding×Event）

```
CorrelationRule {
  id, version
  inputs   : [ {detector_id | event_type, 条件}, ... ]
  join_on  : person_key | counterparty_key
  window   : 時間窓（例: 72h、または同一accounting_period）
  emit     : Finding（axesへの追加寄与つき。クロス成立は逸脱度・切迫性を底上げ）
}
```

例（粉飾・改訂版）：`journal.post`の外れ値Finding × 同一person_keyの`message.delete`メタイベント急増（同一period内）→ クロスFinding。**Slack本文は読まない。削除・編集の頻度と時刻だけを使う**（Opus版の「チャット本文原則不使用」との矛盾をこの規定で解消）。

例（経費不正）：`expense.apply`と`expense.approve`の間隔 < 60秒 × 申請者と承認者の組織距離（同一上長・相互承認ペア）→ 即時承認・馴れ合い承認候補。

### 5.2 グラフ検知（循環取引）

```
1. 期間ごとに counterparty_key を頂点、受発注/仕訳を辺とする有向グラフを構築
   （辺属性: 合計金額・件数・粗利率）
2. 長さ2〜5の閉路をDFSで列挙（自社を含む閉路を優先）
3. フィルタ: 閉路内の金額類似度（max/min ≤ 1.1 目安、要調整）
           ＋ 粗利の薄さ ＋ 実体シグナル欠如（納品記録・検収の不在）
4. 通過した閉路をFinding化。evidenceに閉路の全辺（元取引ID）を添付
```

- 前提は§3.3の取引先名寄せ。**名寄せ品質が閾値未満のデータではこのDetectorを有効化しない**（誤名寄せ→冤罪的な誤検知が最悪シナリオ）。

## 6. LLMの位置づけ（補助要約のみ・隔離）

- 役割：**検知確定後のCaseを承認者向けに要約するだけ**。検知・スコア・根拠生成には一切関与しない。学習にデータを渡さない契約のモデルのみ使用（現行はaik-workのGrok補助要約と同型）。
- **プロンプトインジェクション対策（新規）**：evidenceにはファイル名・申請コメント等、**行為者が制御可能なテキスト**が含まれ得る。要約器はこれらを命令でなくデータとして扱う構造化入力にし、要約出力には「AI生成の参考情報」ラベルを固定表示。**要約なしでも原本evidenceに常にアクセスできるUI**とし、要約が判断の唯一の根拠にならないようにする。
- LLM出力はCaseレコードに書き戻さない（表示時生成・キャッシュのみ）。スコアにも状態遷移にも影響させない。

- 経費不正の代表ルール擬似コード（実装粒度の例）：

```
rule 自己承認: apply.actor == approve.actor → dev+=1.0, evidence=両イベント

rule 重複: 領収書ハッシュ or (金額×日付×店舗名正規化) の一致 → dev+=0.9
```

## 8. 誤検知の運用ループ（新章）

検知エンジンは作って終わりではなく、**FPを食べて痩せる仕組み**が本体。

### 8.1 トリアージ状態機械

```
新規 → 確認中 → 確定(TP) / 誤検知(FP) / 様子見(期限つき)
```

- 状態遷移はすべて操作者・時刻・理由つきで監査ログへ。FPには**理由コード**（データ誤り/名寄せ誤り/ルール過敏/業務上正当）を必須入力にする——理由コードがチューニングの入力になる。

### 8.2 抑制ルール（suppression）

- 「この主体×このルールは業務上正当」を**期限つき**で登録（例：90日。無期限の抑制は作らせない）。期限切れで自動再有効化。抑制の登録・延長も承認操作（ホーリーWF）。

### 8.3 シャドーモード

- 新Detector・新ルールは**2〜4週間サイレント稼働**（Findingは記録するがコンソールに出さない）。分布とヒット数を見てからアラート予算内に閾値を置いて公開。これを標準投入手順にする。

### 8.4 メトリクス

- ルール別 precision proxy＝TP/(TP+FP)。**フロア（例0.2、要合意）を下回ったルールは自動でシャドーへ降格**し、改修まで復帰させない。
- 補助指標：time-to-triage、アラート予算遵守率、名寄せ成功率（ソース別）。これらはmichizaneの「守り」面の運用健全性として可視化。
- **偽陰性（見逃し）**は原理的に測れない。代替として、判明した事案（監査・通報経由）の遡及テスト——「道真はこれを捕れたか」——を四半期ごとに実施し、捕れなかった事案は新ルールの起票源にする。

## 9. 運用モデル（SKYDIV夜間）＋AIガバナンス

- **夜間**：隔離VDI(SKYDIV)の空きデスクトップでAIが**読み取り・下書きのみ**実行（検知バッチ、プロセスマイニング→設計書たたき）。ガードレール＝隔離／allowlist／反復上限／全操作ログ／外部送信・コピー禁止／成果物は指定フォルダのみ。
- **朝**：ホーリーの朝ダイジェストに新規Case（予算内の件数）を差し込み、人がレビュー・確定。**是正の実行はホーリーの既存承認フローを再利用**（道真専用の承認系を新造しない——タツヤ資産の転用でD層の実装量を大幅圧縮）。
- **AIガバナンス（自己監視）**：夜間AIの操作ログ自体をNormalizedEventとして取り込み、AIガバナンスDetector（allowlist逸脱・過剰反復・時間外）で監視。担当＝IRI。
- **率直な指摘**：夜間に広範な読み取り権を持つAIは、それ自体が新たなインサイダー脅威面である。上記ガードレールに加え、**夜間AIのアクセス範囲は「当夜のバッチに必要なデータ契約分のみ」に絞る**（常時全データ読取権を与えない）。

## 10. 統制・プライバシー・法務

- **正しい用語で設計する**：person_keyのハッシュ化は個人情報保護法上の**仮名加工に相当し、匿名加工ではない**。対応表が存在する以上「個人データ」としての管理義務は残る。少人数部署・特定役職では属性だけで再識別可能であり、「匿名だから自由に使える」という前提を全ドキュメントから排除する。
- **目的限定・最小化**：利用目的＝不正・情報漏洩・取引リスクの検知に限定。人事評価・営業成績管理への転用禁止を明文化（転用が始まると従業員の受容が崩れ、プラットフォーム全体が死ぬ）。
- **従業員への周知**：操作ログ・申請データのモニタリングは就業規則・社内規程に根拠を明記し事前周知（モニタリングの適法性の通例要件。労務・法務レビュー要、要確認）。ハラスメントDetectorは通報起点・範囲限定・法務承認を運用条件とし、労使合意を前提にPhase3まで着手しない。
- **アクセス統制**：Case閲覧は監査/管理部門＋AI推進室に限定（@eltes.co.jpゲート＋役割ベース）。person_key復元は監査部門の承認操作のみ（§3.2）。
- **保持期間**：生イベント・Finding・確定Caseで別々に定める（例：生イベント6ヶ月／FP Finding 1年／確定Case 7年——初期案であり監査・法務と要確認）。無期限保持をデフォルトにしない。
- **監査証跡**：スコアのスナップショット（§4.4）、状態遷移ログ（§8.1）、person_key復元ログ、RefData（閾値・重み）変更履歴。これで「なぜあの時この候補が出て、誰がどう判断したか」を完全再現できる。

## 11. 価値可視化（michizane）

- **攻め**：AX診断＋オートクリスタルのSaaS削減実測（Opus版記載の実測値：995h/月・¥35.8M/年）。実測系はこれまで通り。
- **守り＝防いだ損失額（算定ルールを保守化）**：
  - `防いだ損失額 = Σ 確定Caseごとの (想定損失額 × 顕在化確率係数 × 確度係数)`
  - 想定損失額はCaseの実額（不正申請額・持ち出しデータの想定価値等）を根拠つきで記録。係数は監査部門が承認した値のみ使用（初期値は要合意、要確認）。
  - **「実測」（実際に阻止した実額）と「推計」（早期検知による圧縮効果）をダッシュボード上で必ず分離表示**。攻めの実測値と守りの推計値を合算した単一数字を作らない——経営への信頼性はここで決まる。
- 「守った円は攻めの再投資原資」のストーリーは維持。ただし数字の根拠区分を落とさないこと。

## 13. ロードマップ（完了基準つき）

**クリティカルパスは検知ロジックではなくデータ供給と権限**。各Phaseは「使えるデータ」を起点に切る。

道真の展開は**e-RealtyとEltes（タツヤ）の二系統**で進む。e-Realtyはホーリー（賃貸管理システム）内にOBIC会計データを既に持つため、粉飾・循環取引を会計連携待ちなしで前倒しできる。Eltes側の会計はオートクリスタル経由の奉行/MF連携が別途必要（要確認）。

- **Phase 1（〜3ヶ月目安）**：経費不正Detector実データ化＋取引先OSINT。**＋（e-Realty系統）ホーリー内OBIC仕訳の仮名化エクスポートを同じDataContract手順に載せ、粉飾・循環取引のコールドスタート（分布取得・シャドー投入）まで前倒しで進める**。
  - 段取り：ホーリーの仮名化エクスポート（DataContract v1締結。e-Realtyは申請/承認/QAに仕訳・取引先・物件マスタを含める）→ コールドスタート閾値設定 → シャドー2週 → 監査部門トリアージ開始。
  - **Exit criteria**：実データ1ヶ月分を処理／朝ダイジェスト運用が監査部門で回る／ルール別precision proxyの計測が開始されている／防いだ損失額の算定ルールに監査部門が合意。（e-Realty前倒し分は「シャドー投入まで」を目標とし、公開はPhase2の名寄せ品質ゲート通過後。）
- **Phase 2**：粉飾・循環取引の本番公開（e-Realty先行→Eltes追随）＋SKYDIV夜間バッチ＋AIガバナンスDetector。
  - 前提（Eltes系統）：会計(奉行/MF)・SFA連携（オートクリスタルSaaSアダプタ経由）、SFA側はsfa-analysis-platformのBQ連携が候補経路。両系統共通：**取引先名寄せ品質が基準到達**（法人番号付与率等の目標値を事前設定、要確認）。
  - **Exit criteria**：夜間バッチが人手ゼロで安定稼働（連続20営業日目安）／循環取引Detectorがシャドーを卒業（まずe-Realty）／夜間AI自身のガバナンスFindingがゼロでない（＝監視が実際に機能している証拠）こと。
- **Phase 3**：ハラスメント（法務・労使合意後）／全社横断クロス検知／AIガバナンス常時化。
  - **着手条件（Exitでなく入口条件）**：就業規則・規程整備完了＋労使合意＋通報起点フローの法務承認。条件未達なら着手しない。

## 14. 弱点・リスク・未解決論点（率直に）

| # | 論点 | リスク | 対処案 |
|---|---|---|---|
| 1 | **属人化** | 設計・実装・運用が実質1名（AI推進室）に集中。不在で停止 | Phase1完了時点で監査部門に運用（トリアージ・閾値承認）を移管。設計書・Runbookを成果物に含める |
| 2 | **データ供給が最大の不確実性** | 会計/SFA/SKYSEAの接続権限・情シス調整が遅延しがち | Phase切りをデータ到達基準にした（§13）。手動エクスポートで先行し、接続待ちで全体を止めない |
| 3 | **OSINTの名誉毀損・同名誤認** | 同名他社のネガ情報を取引先に紐づける事故 | 法人番号・所在地での確証がないヒットは「未確証」区分で人に渡す。OSINT Findingは社外に出さない内部限定 |
| 4 | **名寄せ誤りによる冤罪的誤検知** | 循環取引・粉飾で最も深刻 | 名寄せ品質ゲート（§5.2）。曖昧一致は人の確定必須（§3.3） |
| 5 | **夜間AI自身の脅威化** | 広範な読取権を持つ自律AIが漏洩経路になる | 当夜必要分のみのアクセス（§9）＋AIガバナンスDetectorで自己監視＋外部送信遮断 |
| 6 | **監視による組織文化の毀損** | 「監視される会社」化で従業員の信頼を失う | 目的限定の明文化・転用禁止（§10）・検知対象は行為パターンであり人物評価でないことの周知 |
| 7 | **偽陰性が測れない** | 「道真があるから安心」という誤った安全感 | 判明事案の遡及テストを四半期実施（§8.4）。経営報告に「見逃しは測定不能」と正直に記載 |
| 8 | 個人事業主・海外取引先の名寄せ | counterparty_keyの設計外 | Phase2で別途設計（未解決） |
| 9 | 閾値・係数の初期値 | 本書の数値例は多くが仮置き | 「要確認」明記箇所を監査部門・法務との合意で確定してから運用開始 |
| 10 | ホーリー自体の障害・改ざん | データ源かつ承認系の単一依存 | ホーリー側監査ログ（qa_results）とのクロスチェック、エクスポートのハッシュ検証（Phase2で設計） |
